# 要件定義書：koko-dayo（物件位置推定システム）

## 1. サービス概要
*   **名称：** koko-dayo
*   **コンセプト：** 不動産サイトの「施設名」と「距離」のリストから、物件の真の所在地を道路網ベースで特定し、地図上に可視化する。
*   **ターゲット：** 正確な物件位置を効率的に特定し、現地確認や周辺環境の調査（ストリートビュー等）を行いたいユーザー。

---

## 2. 構成・アーキテクチャ方針（コスト最適化戦略）
「なるべく無償枠・低コスト」を実現するため、以下の構成を採用する。

| コンポーネント | 採用技術・サービス | 選定理由 |
| :--- | :--- | :--- |
| **テキスト解析** | Gemini 1.5 Flash | 高い抽出精度と、寛大な無料枠（15回/分）を両立。 |
| **施設検索 (Geocoding)** | Mapbox Search API | 月間10万件まで無料。Google Places APIの代替。 |
| **道路網計算 (Routing)** | OSRM (OpenStreetMap Service) | セルフホスト可能で、API制限なく「道のり」を計算可能。 |
| **地図描画** | Leaflet / OpenStreetMap | ライブラリ使用料無料で、カスタマイズ性が高い。 |
| **インフラ** | Vercel / Docker (local/VPS) | 個人開発での無料枠・低コスト運用に適している。 |

---

## 3. 機能要件

### 3-1. データ入力・抽出モジュール
*   **自動抽出：** ユーザーが不動産サイトからコピーしたテキストを貼り付けると、Gemini APIが以下の項目をJSON形式で抽出する。
    *   施設正式名称（例：「セブンイレブン 練馬谷原6丁目店」）
    *   道路距離（メートル表記）
    *   徒歩分数
*   **手動入力・補正：** 抽出されたリストの編集、追加、削除。および手動での新規施設入力。

### 3-2. ジオコーディングエンジン
*   **座標変換：** Mapbox Search APIを用いて、施設名を緯度・経度に変換。
*   **検索地域最適化：** テキスト内から住所（市区町村）を抽出し、検索クエリに含めることで同名施設の重複による誤認識を防止。

### 3-3. 高精度立地推定ロジック（コア機能）
直線距離ではなく、道路網（OSRM）を用いた「グリッド・スコアリング法」を採用。
1.  **グリッド生成：** 候補エリアを約10m〜20m間隔のメッシュ（点）に分割。
2.  **道路距離マトリクス計算：** OSRMの `Table` APIを用い、各グリッド点から全施設への**実際の徒歩ルート距離**を一括計算。
3.  **誤差評価 (MAE)：** 各グリッド点において、`|OSRM算出距離 - 入力表記距離|` の総和を計算。
4.  **確率分布変換：** 誤差が小さい地点ほどスコアが高くなるよう変換し、ヒートマップ用のデータを生成。

### 3-4. 地図可視化・照合機能
*   **ヒートマップ表示：** 推定エリアを地図上に可視化。
*   **建物特定サポート：** 
    *   推定地点をクリックすると、その場所のGoogleストリートビューを別ウィンドウまたは埋め込みで起動。
    *   不動産サイト掲載の外観写真と目視で比較しやすいUI。

---

## 4. 画面設計イメージ

1.  **入力画面 (Top)：** テキスト貼り付け用のテキストエリアと「解析開始」ボタン。
2.  **確認画面 (Review)：** 抽出された施設リストの確認・修正。
3.  **結果画面 (Map)：** 
    *   左側：メインの地図（ヒートマップ表示）。
    *   右側：推定地点の情報、付近の住所、ストリートビューへのリンク。

---

## 5. 技術的課題と対策（リスク管理）

| 課題 | 対策 |
| :--- | :--- |
| **徒歩分数の定義差** | 「80m=1分」を基本とし、距離(m)の記載がある場合は常に距離を優先して計算する。 |
| **施設特定（駅等）** | 駅は出入口（改札）によって座標が大きく変わるため、Mapbox APIの `poi` カテゴリを活用して可能な限り正確なポイントを取得する。 |
| **計算負荷** | 広範囲のグリッド計算は重いため、まずは直線距離で絞り込んだ後、その周辺のみをOSRMで精査する2段階評価を行う。 |

---

## 6. 開発ロードマップ

*   **Phase 1 (Proof of Concept)：** Pythonによるロジック検証スクリプトの作成（OSRMでの計算精度確認）。
*   **Phase 2 (MVP)：** FastAPI + React による基本機能（抽出・計算・表示）の実装。
*   **Phase 3 (Optimization)：** ヒートマップ表示の洗練、OSRMの自前運用によるコストゼロ化。
